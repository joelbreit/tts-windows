1. Use-Case Context

This app is a personal reading assistant triggered on-demand via a global hotkey. Key characteristics that drive the evaluation:

  • Trigger pattern: User selects text, presses Win+Alt+R — they expect speech to start quickly (<500ms perceived).
  • Text length: Anything from a URL slug to multi-paragraph articles. Streaming matters for long texts.
  • Volume estimate: Personal use at active reading pace (~150 wpm, ~800 chars/min). A moderate session of 30–60 min/day yields roughly 50k–150k chars/month.
  • Context: Windows desktop, always-on internet assumed acceptable. Fully offline is a nice-to-have, not a hard requirement (that constraint was in the original spec, but has since been relaxed).
  • Stack: .NET 10 WPF. Existing TtsService.cs wraps WinRT SpeechSynthesizer — the new provider would replace or extend that class.
Why the local "Natural" voices can't be used (verified 2026-06-08)
Installing a Windows 11 Natural / Natural HD voice (e.g. Ava, Aria) does not make it usable by this app — and this is by design, not a bug. These voices ship as Appx packages (MicrosoftWindows.Voice.en-US.AvaHD.1) carrying their own private neural engine (~580 MB of encoder/decoder/vocoder .bin models + a COM CLSID). They register only via the Narrator-private AppExtension contract com.microsoft.voice.model.1 — their declared token hive (HKLM\SOFTWARE\Microsoft\Speech Server\v11.0) doesn't even exist on the machine, and their COM engine isn't in HKEY_CLASSES_ROOT. As a result they are invisible to all three public TTS surfaces: Windows.Media.SpeechSynthesis (OneCore, what this app uses), System.Speech (SAPI5), and any OneCore token store. Microsoft reserves them for Narrator / the accessibility stack. Conclusion: a cloud or local-model TTS provider is the only realistic path to better voices. The good news — the same Ava voice is available as en-US-AvaNeural via Azure below.

2. Evaluation Criteria

Ranked by importance for this use case:

  1. Voice quality / naturalness — The whole reason to upgrade from David/Zira.
  2. First-audio latency (TTFB) — Time from hotkey press to first audible word. Streaming APIs can start playing before the full text is synthesized.
  3. Price — At ~50–150k chars/month for personal use.
  4. .NET / C# integration — How easily it slots into the existing WPF app.
  5. Streaming support — Critical for long texts; degrades gracefully for short ones.
  6. Voice variety & control — Ability to pick a preferred voice, adjust prosody, etc.

Kokoro is an 82M-parameter model that punches well above its weight — quality is close to cloud neural voices. It's available as ONNX weights and can be run via Microsoft.ML.OnnxRuntime directly in .NET without any subprocess. The downsides: CPU-only inference is noticeably slower than Piper (~200–600ms vs ~50–150ms for short texts), the integration is more complex (you need to handle tokenization and audio decoding), and the 300MB model bundle is larger. What about Windows "Natural" Voices? (Don't count on them)
It's tempting to assume the free Windows 11 Natural voices show up in SpeechSynthesizer.AllVoices — they do not. As verified on this machine (see the boxed note at the top), the Narrator Natural/Natural HD voices are walled off from every public TTS API by design. Installing more of them will not help this app. The legacy (Natural)-suffixed OneCore neural voices from older Windows builds could appear via this API, but they are increasingly being replaced by the Narrator-only HD packages, so this is not a path to rely on.

7. Integration Notes (.NET / C#)

The existing TtsService.cs has a clean SpeakAsync(string text, VoiceInformation voice, double rate) interface. The suggested pattern is to introduce a ITtsProvider interface so the app can swap providers without rewiring the ViewModel.

Azure AI Speech — Streaming Integration Sketch

NuGet: Microsoft.CognitiveServices.Speech
// In TtsService or a new AzureTtsProvider class
using Microsoft.CognitiveServices.Speech;
using Microsoft.CognitiveServices.Speech.Audio;

var config = SpeechConfig.FromSubscription("<key>", "eastus");
config.SpeechSynthesisVoiceName = "en-US-JennyNeural";

// Streams directly to default audio device — no MediaPlayer wiring needed
using var synth = new SpeechSynthesizer(config);
var result = await synth.SpeakTextAsync(text);

// For streaming (first audio before full synthesis):
synth.SynthesisStarted += (_, e) => /* start playback */;
var stream = AudioDataStream.FromResult(result);

OpenAI TTS — Streaming via HttpClient

No NuGet needed — standard HttpClient
using var http = new HttpClient();
http.DefaultRequestHeaders.Authorization =
    new AuthenticationHeaderValue("Bearer", "<key>");

var body = JsonSerializer.Serialize(new {
    model = "tts-1",
    input = text,
    voice = "nova",
    response_format = "mp3"
});
var response = await http.PostAsync(
    "https://api.openai.com/v1/audio/speech",
    new StringContent(body, Encoding.UTF8, "application/json"));

// Stream the mp3 bytes directly into MediaPlayer via a custom IRandomAccessStream wrapper
var audioStream = await response.Content.ReadAsStreamAsync();

Piper TTS — Subprocess Integration Sketch

Bundle piper.exe + model .onnx alongside your app
var psi = new ProcessStartInfo("piper.exe") {
    Arguments = $"--model en_US-ryan-medium.onnx --output-raw",
    RedirectStandardInput  = true,
    RedirectStandardOutput = true,
    UseShellExecute = false,
    CreateNoWindow  = true
};
var proc = Process.Start(psi)!;
await proc.StandardInput.WriteAsync(text);
proc.StandardInput.Close();

// Raw PCM 16-bit 22050 Hz mono out — wrap in WAV header or feed directly to NAudio/WaveOut
var rawAudio = proc.StandardOutput.BaseStream;

Key Architectural Note

Cloud APIs return MP3 or OGG audio bytes, not a WinRT SpeechSynthesisStream. The existing MediaPlayer setup can still handle this — feed the bytes into a Windows.Storage.Streams.InMemoryRandomAccessStream and pass it to MediaSource.CreateFromStream(), the same pattern already in your code. For streaming playback, you'll need to buffer chunks as they arrive and push into a custom IRandomAccessStream.

The speed/rate control currently uses PlaybackSession.PlaybackRate on the WinRT MediaPlayer — this will continue to work regardless of TTS provider since it operates on the audio playback layer, not the synthesis layer.
